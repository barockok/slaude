## v0.16.0 тАФ DM allowlist, reliable cron, OAuth discovery that actually connects

A small, sharp follow-up to v0.15.0's per-user OAuth work. Three field-driven fixes: `/mcp connect` now discovers OAuth on servers that don't emit a challenge header (the common case тАФ it was 404'ing on workbench); cron jobs stop silently skipping themselves when a human is active; and DMs are no longer manager-only тАФ you can whitelist teammates to message slaude 1:1.

### Features

- **Soul: DM allowlist.** New `dmAllowedUsers` field on `SoulData`, parsed from a `## DM allowlist` section in SOUL.md exactly like `manager` / `trustedChannels` / `blockedUsers` тАФ LLM extraction with verbatim grounding (an id not present in the persona is rejected; regex fallback never fills it). The engagement gate now admits a DM from any whitelisted user on top of manager/backup. Scoped to DMs only, and grants *chat engagement only* тАФ admin commands (cron / ignore / ingest / 1on1) still require manager/backup or an approver. Documented in the starter persona template.

### Fixes

- **MCP OAuth: RFC 9728 well-known discovery fallback.** `/mcp connect <server>` failed with `did not advertise resource_metadata (status 404)` against servers that support OAuth but don't return a `WWW-Authenticate` challenge on a bare GET (e.g. workbench тАФ its MCP endpoint 404s on GET yet serves protected-resource metadata at the well-known URL). `discover()` now falls back to `/.well-known/oauth-protected-resource` derived from the server URL тАФ path-aware variant first, then root тАФ before giving up. The 401-challenge path stays preferred. Host + protocol are pinned to the trusted `mcp.json` config; only the path is derived (no SSRF surface).
- **Cron: fire on schedule even when the thread/channel session is live.** Jobs were suppressed with `skipped: session live` whenever a human was active in the target thread/channel, so a busy channel could silently starve its own digest/reminder тАФ exactly where it's most wanted. Dropped the `isLive` guard: cron runs by default regardless of activity. Same-job re-entry is still guarded by the `#running` set.

### Internal

- New gate test (`tests/gateway/sim/dm-allowlist.test.ts`) drives admit/drop DM paths through the real gateway; soul-extract tests cover grounded + ungrounded `dmAllowedUsers`.
- Security review across all three changes: no HIGH/MEDIUM findings (DM scoping correct, discovery fallback is same-origin/path-only, cron crosses no auth boundary).
- 741 tests pass, tsc clean.
