## v0.17.0 — global `/mcp` connect

`/mcp connect` shipped (v0.15.0) as initiator-only inside a `/1on1` lock: it wires an OAuth HTTP MCP server *as the locked-in user*, writing their per-user config home. There was no way to connect a server for the **agent's own shared identity** — the one every unlocked session uses — short of the operator hand-seeding `.credentials.json`. This release adds that path.

### Features

- **MCP: global `/mcp connect`.** Running `/mcp connect <server>` outside any `/1on1` lock now connects the agent's shared identity — the token is written to the agent's own config dir (`CLAUDE_CONFIG_DIR`), which every unlocked session inherits. Manager/backup only, since a global connect mints credentials the agent uses for everyone (operator-tier, same trust as editing `mcp.json`). Inside a `/1on1` lock the behavior is unchanged: initiator-scoped, written to `$SLAUDE_HOME/oauth/<userId>`. Scope is chosen automatically by lock state and carried through the connect button and paste-back flow, so neither can be replayed across the boundary. See [finding](../findings/2026-06-10-mcp-global-connect.md).

### Internal

- `ConnectScope` (`initiator` | `global`) threaded through the `/mcp` slash handler, `connectServer`, `persistTokens` (dir selection), the button handler (`pendingMcp`), and paste-back (`pendingPaste`).
- Button re-verifies on click per scope: initiator → still owns the lock; global → no lock present **and** still manager/backup (fresh `soulData()`). Paste-back binds on the `channel:thread:userId` key (signed inbound id), not the lock.
- Security review: no findings — manager-only gate enforced on all entry points, no scope replay, no path traversal (userId never in the global dir), disjoint dirs (no token confusion).
- 747 tests pass, tsc clean.
