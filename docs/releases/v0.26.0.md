## v0.26.0 — disengaged threads stay in-session via a suppress hook

A feature release that changes *how* a disengaged thread is handled: instead of
dropping every message after a user turns to a colleague and reconstructing the
gap on re-`@mention`, the gap is now kept **inside the session** as real
conversation history. The model simply doesn't respond while disengaged. Full
writeup: [2026-06-16 reengage-hook-suppress](../findings/2026-06-16-reengage-hook-suppress.md).

### Features

- **Engagement: keep disengaged threads in-session, suppress the reply.**
  Previously the gateway gate *dropped* every plain message after a thread
  disengaged (a user `@mentions` a colleague), so on re-`@mention` the agent
  resumed blind to everything said in the gap. The earlier approach rebuilt that
  gap by re-fetching the thread from Slack and injecting a synthetic preamble —
  heavy: a Slack round-trip, name resolution, recency caps, a truncation story.
  This release retains the gap instead of reconstructing it. Disengaged messages
  now flow into the session so the transcript stays continuously populated, and a
  `UserPromptSubmit` hook stops the model from *responding* while disengaged. On
  re-engage there is nothing to backfill — the context is already there.
  (`src/agent/manager.ts`, `src/gateway/core/gateway.ts`)

- **The load-bearing detail: `continue:false`, not `decision:"block"`.** A spike
  against the pinned SDK settled which hook return achieves "persist but don't
  process": `decision:"block"` discards the prompt *before* it is written to the
  transcript (no jsonl line), so the gap would be lost. `continue:false`
  enqueues and persists the user turn, then halts before generation — exactly
  "session stays populated, model never processes it", at ~0 output tokens.
  `disengagedHookDecision(row)` is pure and exported; the hook reads the session
  row **live** via `Sessions.findById`, so a re-`@mention` takes effect on the
  very next message. A colleague-mention in a thread with *no* session is still
  dropped — slaude never spins one up for an unrelated thread.

### Internal

- **`suppress` routing.** The engagement gate routes disengaged messages via
  `handleMessage(args, {suppress:true})`. `suppress` skips all Slack-visible
  feedback (no 👀, no "thinking…", no ✅ on `done`) and swaps the reply directive
  for a "recorded for context only — do NOT reply" note.
- **Metric.** `slaude_disengaged_suppressed_total` counts suppressed turns
  (`src/metrics.ts`).

### Tests

- `tests/agent/disengage-hook.test.ts` — the pure decision (engaged /
  disengaged / no-row) plus an explicit guard that it never emits
  `decision:"block"`.
- `tests/gateway/core/gateway-seam.test.ts` — the engage/disengage durability
  block rewritten to the new contract: disengaged messages are *recorded but
  suppressed*, never processed; a no-session colleague-mention is dropped;
  disengage survives a restart; re-`@mention` re-engages. Patch coverage 98.03%
  (within the codecov threshold).

### Notes

- The user-facing guarantee is unchanged from the earlier durability work —
  slaude does not reply after you turn to a colleague. What changed is the
  mechanism: the gap is retained as genuine session history instead of being
  dropped and later reconstructed.
