## v0.14.0 ‚Äî contextual connections, a Slack-free simulation harness, and /1on1 mode

Three substantial features land together. The headline is the **simulation gateway**: slaude's whole Slack surface ‚Äî engagement, channel-mode, approvals, slash-command authz ‚Äî can now be exercised in CI and a terminal REPL with no Slack workspace, running the *same* gateway code as production. That harness was then used to develop and prove the other two features (`/1on1` mode and the connect-broker flag) without touching Slack. The **contextual per-user MCP connections** broker also shipped but is **off by default** ‚Äî `/1on1` mode is the per-thread feature enabled out of the box.

### Features

- **Contextual per-user MCP connections (`slaude_connect`).** Ephemeral, thread-scoped service connections (e.g. Jira) owned by the *requesting* Slack user, not a shared bot identity. A stable in-process broker MCP fronts lazily-spawned vendor MCP subprocesses (pool keyed by connection id, idle reaper) and proxies via fixed generic tools (`mcp_call`, `mcp_describe`, `connect`, `connections_list`, `connections_revoke`) so the SDK tool list never changes. Credentials are AES-256-GCM-encrypted at rest (per-row nonce, AAD-bound to the connection id; TTL + reaper). Cross-user borrow requires the owner's per-thread, revocable approval (reusing the Slack approval gate, targeted at the owner). Caller identity is in-band (`on_behalf_of`) and read live server-side, never snapshotted. **Off by default** ‚Äî gated behind `SLAUDE_ENABLE_CONNECT_BROKER` *and* `SLAUDE_ENCRYPTION_KEY` (an explicit switch decoupled from the key), so a default deployment exposes no connection tools. CDP login host + vendor child spawn are deploy-time seams behind tested interfaces. See [finding](../findings/2026-05-29-contextual-mcp-connections.md), [design](../superpowers/specs/2026-05-29-contextual-mcp-connections-design.md), [ops](../connect-broker-login.md).

- **Simulation gateway (`bun sim`).** Verify every Slack-dependent behavior with no real Slack. The Slack adapter was inverted onto a `Transport` port ‚Äî `createSlackApp(agent)` became `createGateway(agent, transport)` ‚Äî so production binds bolt while simulation binds an in-memory `SimTransport`. Bolt's `App` satisfies the port structurally, so the production path is untouched; consistency between sim and prod is guaranteed by construction. A `StubAgent` drives turns by calling the *real* exported MCP handlers (`slackHandlers`, `brokerHandlers`) through a narrow `__sessionCtx` seam, so the approval-gate and the connect-broker run for real without Claude. Ships YAML scenario transcripts (a CI gate via `bun sim run`) and a scenario-first REPL (`bun sim`) sharing one `SimSession` engine, with built-in role/channel presets. See [finding](../findings/2026-05-29-simulation-gateway.md), [design](../superpowers/specs/2026-05-29-simulation-gateway-design.md).

- **`/1on1` mode (per-thread engagement lock).** `/1on1` locks a thread so slaude listens only to the initiator + manager/backup; everyone else is silently dropped (`one_on_one`). `/1on1 off` (initiator or manager) releases. Persistent in sqlite (`one_on_one_locks`), survives restart. The gate sits in `handleMessage` after the channel-mode gate (overriding "anyone can chat" in trusted/allowed channels) and before slash parsing (so a non-allowed user can't `/1on1 off` to hijack a lock). Approvers are unaffected ‚Äî button clicks go through the approval-gate action handler, independent of chat engagement. Proven by three simulation transcripts. See [finding](../findings/2026-05-29-one-on-one-mode.md), [design](../superpowers/specs/2026-05-29-one-on-one-mode-design.md).

### Fixes

- **Sim StubAgent deadlock.** `drain()` awaited the behavior promise, which hung forever on behaviors that park on an approval decision (resolved only by a later button click). Now drains microtasks/one macrotask without awaiting parked work.
- **Soul fixture in CI.** The soul regex fallback only fills `approvers` (manager/channels need the LLM), so the sim now injects structured `SoulData` via the existing production `setSoulData()` instead of a new extract seam; uses valid Slack-format IDs.

### Internal

- **Transport-port refactor** (`src/gateway/core/{transport,gateway}.ts`): the 843-line `adapter.ts` body moved verbatim behind the port; `adapter.ts` is now an 8-line bolt-binding re-export, with the gate constructors (`ApprovalGate`/`PermissionGate`) taking the port instead of a bolt `App`.
- **Sim engine `thread:` field** ‚Äî opt-in shared `thread_ts` on send steps so thread-scoped behaviors (like `/1on1`) model one real Slack thread; omitting it preserves per-send-thread behavior.
- **Coverage:** scenario transcripts lift `gateway.ts` to ~83% line; added `codecov.yml` making the project status *informational* (the sim loads the real `AgentManager` SDK loop + bolt transport into the denominator ‚Äî integration-only code uncoverable without live services), while patch coverage stays enforced.
