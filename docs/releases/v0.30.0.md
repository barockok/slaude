## v0.30.0 â€” channel-scoped souls, a remote brain process, and agent-as-itself on Slack

This release widens how a single deploy adapts to its surroundings. The soul can
now speak differently per channel, the brain (KB engine) can run as its own
OAuth-protected process instead of in-line, the agent can post on Slack as its own
user account rather than the app bot, and cron jobs gained a real lifecycle. All
four are opt-in and default to the prior behavior â€” existing deploys are unchanged.

### Features

- **Soul: channel-specific mandate & approvers (#62).** `SOUL.md` can now carry
  per-channel `## Channel <#Cxxx>` blocks, each with its own `### Mandate` and
  `### Approvers`. A new `effectiveSoulForChannel` resolver applies them with
  *replace* semantics â€” inside an overridden channel the channel's approver list is
  the only approver source for the approval gate and approver-based admin auth, and
  the channel mandate is injected as an authoritative `<channel-mandate>` directive
  that supersedes the global one. The grounding check extends to channel ids and
  per-channel approver ids, so the extractor can't widen the allowlist. Crucially,
  manager/backup are **always retained as catchall approvers** in every channel â€” a
  channel block can add approvers but never lock the operator out. Any resolution
  failure falls back to the global view so the gates never break. See
  `docs/findings/2026-06-20-channel-soul-overrides.md`.

- **Brain: gbrain as a configurable remote OAuth MCP process (#61).** The KB engine
  behind `slaude_kb` can now run as a separate, OAuth-protected MCP process,
  selected with `SLAUDE_BRAIN_MODE=local|remote` (default **local** â€” identical to
  the historical in-process brain). A `BrainBackend` seam routes `brainCall` /
  `brainAdminCall` to a `LocalBackend` (in-process) or a `RemoteBackend` (an OAuth'd
  MCP client to `slaude brain-server`). Scope resolution and write-gating stay in
  slaude; the remote server is a dumb engine exposing two tools (`brain_op` /
  `brain_admin_op`). Auth is a Keycloak-first JWT resource-guard (jose JWKS, RFC 9728
  protected-resource metadata), fail-closed when issuer/audience are unset, with
  `slaude brain connect` reusing the shared OAuth loopback. See
  `docs/findings/2026-06-20-brain-remote-mcp.md`.

- **Slack: opt-in post-as-user via xoxp token (#11).** With `SLACK_POST_AS_USER=true`
  (plus an existing `SLACK_USER_TOKEN`), the agent's replies/edits/reactions/uploads
  go out as its own real Slack account instead of the app bot â€” so a deploy no longer
  maintains two visible identities. The bot token stays for Socket Mode events and
  gate interactivity. Default off; existing deploys that set `SLACK_USER_TOKEN` for
  presence only are unchanged. Guards the self-echo loop that real-user posts (no
  `bot_id`) would otherwise cause. See
  `docs/findings/2026-06-09-post-as-slack-user.md`.

- **Cron: pause/resume/edit lifecycle (#50).** Scheduled jobs gained a real
  lifecycle â€” `/cron pause|resume|edit` (and `/cron-*` aliases) plus runtime MCP
  tools (`pause_cron_job` / `resume_cron_job` / `edit_cron_job`). A `paused` state is
  honored by the scheduler (paused jobs skip unless manually queued), and lifecycle
  actions accept short id prefixes. Previously a job could only be created, listed,
  or removed â€” changing one meant delete-and-recreate. See
  `docs/findings/2026-06-08-cron-on-channel.md`.
