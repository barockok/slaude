## v0.25.1 — status line never leaks secrets or filesystem paths

A patch release hardening the live "is thinking…" Slack status indicator
(`assistant.threads.setStatus`) against leaking sensitive data.

### Fixes

- **Gateway: the status line no longer mirrors raw tool arguments.** The status
  renderer inlined tool input — the `Bash` branch showed the first 50 chars of
  the command — so a secret on the command line (`curl -H "Authorization: Bearer
  …"`, `TOKEN=… cmd`) leaked into the broadcast status, and absolute paths in
  commands/args exposed filesystem structure outside the workspace. The renderer
  is extracted to `src/gateway/core/status-text.ts` and made safe **by
  construction** (fully deterministic — switch / string ops / regex, no model):
  - **Bash** → verb + program name only (`running \`curl\``); args are never
    emitted.
  - **File tools** → basename for absolute paths (never an absolute /
    outside-workspace path), tail segments for relative paths.
  - **WebFetch** → host only (drops path + query, e.g. `?api_key=…`).
  - **`redactSecrets()`** — a central regex backstop over the final string for the
    few branches that still echo user text (Grep pattern, WebSearch query):
    Authorization/Bearer, `api_key|token|secret|password=`, and known key shapes
    (`sk-`, `gho_`/`ghp_`, `xox[bap]-`, `glpat-`, `AKIA`, `AIza`, JWT) → `••••`.

  The primary guarantee is structural — args/paths/queries are never rendered, so
  there is nothing to leak regardless of secret shape; redaction is a secondary
  net since regexes can miss novel formats.

### Internal

- New `tests/status-text.test.ts` (44 tests, 100% patch coverage) pins the
  invariant: secret-laden commands and outside-workspace paths render only the
  safe verb / program / basename / host. Scope is the ephemeral status text only —
  no change to logs, tool I/O, or agent behavior.
