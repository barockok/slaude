## v0.23.2 — cron channel-target posts to channel root

A patch release fixing a cron delivery bug. A UAT case (agent "Maria")
created scheduled jobs with `target: channel`, confirmed in the job JSONL —
`{"cron_expr": "*/15 * * * *", "target": "channel", ...}` — yet every fire landed
threaded under the originating thread instead of at the channel root.

### Fixes

- **Cron: `target=channel` jobs now post at channel root, not the thread.** The
  cron execution context sets `postTarget=channel`, and the legacy
  `slaude_slack` reply path honored it (`thread_ts` omitted when channel-target).
  But the live `slaude_surface` path reads `thread_ts` from the `SessionBinding`,
  and `bindingFor()` built `threadRef` from `ctx.threadTs` unconditionally —
  dropping `postTarget` entirely. So every surface `reply`/`upload` threaded
  under `job.slackThreadTs`. Fixed at the single seam: `bindingFor()` now yields
  `threadRef=undefined` when `postTarget=channel`, so `reply`, `upload`, and
  `getHistory` all post at channel root. Normal inbound sessions never set
  `postTarget`, so their threading is unchanged.
  (`src/gateway/core/gateway.ts`)

### Internal

- Regression test: `tests/gateway/core/binding-target.test.ts` — channel target
  drops `threadRef`; thread target and unset target both keep it. Full suite
  green (992 pass).
