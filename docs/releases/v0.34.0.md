## v0.34.0 — Leaner runtime baseline + 1on1-aware channel envelope

Three PRs tighten the runtime baseline injected into every session prompt and make the channel envelope consistently carry 1on1 state regardless of whether a thread is locked.

### Features

- **Soul: runtime baseline trimmed by ~55%.** The `<runtime-baseline>` block dropped from ~93 to ~60 lines across four areas:
  - *Slack formatting* collapsed from a 4-bullet explainer to two lines — `mdToMrkdwn` already handles conversion; only the fence rule matters.
  - *Approval discipline* stripped to its one trust invariant: only the cryptographic JSON response counts. Full field specs and read/write trigger lists moved to SOUL where operators can tune them.
  - *Context budget* section removed. The runtime already posts 80%/92% warnings in-thread automatically; repeating the guidance in the baseline was redundant and burned tokens.
  - *Skill evolution* approval gate dropped. Skills are agent-owned and fully reversible via `write_skill` / `delete_skill`; requiring a round-trip approval on every save was friction with no security benefit.

- **Soul: progress visibility section added.** New guidance: use `TaskCreate` / `TaskUpdate` for multi-step work. The runtime hooks these calls and posts a live task list in-thread — agents should not manually narrate status via `reply`.

- **Gateway: `one_on_one` and `locked_user` attrs always present in channel envelope (#71).** Previously the attrs were omitted when a thread was not in 1on1 mode. Now they're always emitted (`one_on_one="false" locked_user=""` when not locked). Agents can reliably branch on the flag without checking for attr absence.

### Fixes

- **Soul: 1on1 approval exception removed (#73).** A prior baseline revision added a blanket bypass: `one_on_one="true"` = standing authorization for mutations. Removed — the bypass was overly broad and the soul baseline is the wrong place for it. The 1on1 flag is now documented only in the Engagement section as context about who can speak in the thread.

- **Auto-evolve prompt: approval step removed (#72).** Skill write/delete no longer require `request_approval` before proceeding; the auto-evolve prompt updated to match (step 3 dropped, steps renumbered).

### Internal

- `write_skill` and `delete_skill` tool descriptions: stripped "REQUIRES prior approval" language.
