## v0.21.0 — Runtime-editable souls, a hardened approval gate, and the 0.3 SDK

This release makes a deployed agent's identity *editable at runtime* — a manager
can grow or shrink the agent's trust/DM/block lists from Slack without a
redeploy — and closes a real approval-bypass hole in the same breath. Under the
hood it moves onto the `claude-agent-sdk` 0.3 line and lands a large unit-test
and CI hardening pass so the suite is trustworthy again.

### Features

- **Soul: runtime ACL overrides without redeploy.** A new `/soul` dispatcher
  parses `trust | allow | dm | block  add | remove` plus `list` / `clear`, gated
  to manager-initiated turns. Overrides persist in a new `soul_overrides` table
  and `soulData()` re-applies them on every read, so the effective persona ACLs
  update live. A `soul_override` MCP surface tool exposes the same mutations to
  the agent, also manager-gated. Previously the trust/DM/block lists were frozen
  in `SOUL.md` and only changeable by editing the file and redeploying.
  See [docs/findings/2026-06-11-soul-runtime-overrides.md](../findings/2026-06-11-soul-runtime-overrides.md).

- **Baseline: upload file artifacts instead of paths.** The runtime baseline now
  instructs the agent that when a tool produces a file (image, PDF, CSV, log
  dump, generated asset) it must upload the file itself via
  `mcp__slaude_surface__upload` rather than replying with a local path or an
  expiring URL — inline content survives link expiry, crosses networks, and
  renders in-thread.

### Fixes

- **Security: close the approval-gate bypass.** The approval discipline in
  `RUNTIME_BASELINE` is tightened so the agent trusts *only* the JSON response
  returned by `mcp__slaude_surface__request_approval`. It must never accept a
  screenshot, a text message, a verbal "I approve", or any other out-of-band
  evidence as a substitute — even an uploaded image claiming approval still
  requires the real tool call and its system response. Closes a
  prompt-injection / social-engineering path to running mutating operations
  without genuine manager approval.

- **Engagement: disengage now survives time and restarts.** Per-thread
  engagement state is persisted instead of living only in memory, so a
  disengage actually sticks across restarts rather than lasting effectively zero
  messages. See [docs/findings/2026-06-11-engagement-disengage-durability.md](../findings/2026-06-11-engagement-disengage-durability.md).

- **/1on1: resume no longer breaks on lock flips.** Locking or unlocking a
  thread flipped the per-initiator config home and made `resume` search the
  wrong transcript directory (cold start at lock, stale context at unlock). The
  transcript home is now shared across lock flips. See
  [docs/findings/2026-06-11-1on1-transcript-sharding.md](../findings/2026-06-11-1on1-transcript-sharding.md).

- **Agent: type the thinking content block.** The 0.3 SDK's `BetaContentBlock`
  union types only text + tool_use, but the model still emits thinking blocks at
  runtime — `#fanout` now widens the block type so the thinking branch
  type-checks instead of failing the build.

### Internal

- **Dependency: `claude-agent-sdk` 0.1.77 → 0.3.173.** Major jump onto the 0.3
  line; the SDK message/content types changed shape (see the thinking-block fix
  above).

- **Tests: coverage 94.89% → 99.26% lines.** ~138 new unit tests across the
  agent session lifecycle, core gateway branches, Slack MCP tools + transport,
  the sim REPL/engine/TUI, and db/knowledge. `AgentManager` went from 20.6% to
  ~100%, and the production Slack transport from 12.5% to 100% (mocked bolt App,
  no network). The SDK is now injected via a namespace dereference so test
  mocks apply to the canonical `manager.ts` rather than a query-string phantom
  module — keeping coverage attribution correct.

- **CI: gate on the failure count, not bun's raw exit code.** bun 1.3.11 exits
  99 for this suite even when every test passes — a runner-internal quirk that
  reproduces with coverage fully disabled and surfaces no rejection. The `ci`
  job now parses the printed failure count: it fails on any real test failure or
  a genuine crash (exit ≠ 0/99) but tolerates the spurious 99. Coverage itself
  stays gated by Codecov. This is the first time the repo's `ci` job can go
  green.
