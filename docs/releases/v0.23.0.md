This release gives slaude a way to explain itself and a way to change models on
the fly. The `how-slaude-works` skill is the agent's own onboarding doc; `/model`
lets a manager retune the engine per Slack thread without a redeploy.

### Features

- **Skills: `how-slaude-works` self-documentation.** A bundled skill that
  describes slaude's architecture, surfaces, and conventions â€” the agent can now
  answer "how do you work?" from a maintained source instead of improvising.
  Bundled skills are boot-seeded into `$SLAUDE_HOME` on startup
  (`src/skills/seed.ts`), so a fresh deploy ships them without manual install.

- **Gateway: `/model` per-thread model switch.** `/model` lists the provider's
  available models (`GET /v1/models`, 5-minute in-memory cache) and shows the
  current one; `/model <id>` swaps the model for *this thread's* session. The
  change is applied live via the SDK `Query.setModel()` and persisted to
  `sessions.model`, so it survives a reaper close / restart. Drop to a cheap
  model for chatter in one thread, bump to Opus for hard work in another.

  - **Validation against the live provider.** The typed id is checked against the
    `/v1/models` list â€” typo protection. Non-Anthropic gateways that don't expose
    `/v1/models` fall back to pass-through with a âš ď¸Ź "couldn't verify" notice, so
    OpenRouter / Z.ai / self-host keep working.
  - **ACL.** Allowed for manager, backup manager, approvers, and `dmAllowedUsers`
    â€” the same authority surface as `/ingest`, widened to the DM allowlist since
    per-thread model choice is low-risk and self-scoped.
