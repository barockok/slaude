## v0.24.0 — brain memoize & recall, made trustworthy

This release closes the **brain-memoize investigation** — a multi-day dig into
why "remember this" didn't reliably round-trip to "recall it later." Three UAT
reports, traced through session JSONL and the live UAT brain, surfaced several
distinct failures across the write and read paths. The full forensic record and
decisions live in
[docs/findings/2026-06-14-brain-memoize-failure.md](../findings/2026-06-14-brain-memoize-failure.md).

The throughline: make the brain the single, opinionated home for knowledge —
one write path (`kb_memoize` → gbrain DB), one read source (gbrain DB), and
honest signals when something doesn't land.

### Features

- **Brain: per-thread, per-writer standing grant for approvals.** A
  non-destructive brain write in a trusted channel cards once; after a human
  approves, that same writer's later writes in the same thread auto-pass for an
  8h window (`SLAUDE_KB_GRANT_TTL_MS`; `0` disables). In-memory only — a process
  restart re-asks. The gate's authorization is untouched: deny/scope/manager
  tiers and destructive ops always card; only the repeat approval *card* is
  suppressed for an already-approved writer. (`src/knowledge/gated-dispatch.ts`,
  `threadKey` in `src/gateway/core/gateway.ts`.)

### Fixes

- **Brain: `kb_think` no longer drops a present page (Mode B′).** The hybrid
  gather could rank a rich, embedded, in-scope page below noisier neighbors and
  then answer confidently from the wrong ones — so the old zero-citation
  fallback never fired. `kb_think` now *always* runs a cross-check `kb_search`
  on a **distilled** keyword query (verbose NL → keywords; proven by a case
  where a tight `kb_search` hit rank 1 where the full-question `kb_think`
  missed) and surfaces any strong hit the synthesis didn't cite via
  `search_fallback`. (`src/knowledge/mcp-tools.ts`.)
- **Brain: actionable write errors (#7a).** A raw `pages_source_id_fkey` /
  `$libdir/vector` string is meaningless to the agent and led it to abandon the
  brain and silently write files. `humanizeBrainError` now maps known brain/
  Postgres errors to "what to do": FK / source-not-ready → "retry once, else
  report it did NOT land — don't fall back to a file or claim it saved"; vector-
  ext fault → "infra degraded, don't work around with files". Unknown errors
  pass through.
- **Brain: no silent success after a failed write (#7b).** The soul baseline now
  forbids papering over a failed `kb_memoize` — a write error means the
  knowledge did not persist; retry or tell the user plainly, never reply
  "saved!" after an error, never fall back to files.
- **Brain: one opinionated write path in the soul prompt.** Dropped the stale
  `raw/` → `/ingest` write guidance that produced the "a memoized page needs
  `/ingest`" misinformation. `kb_memoize` upserts straight into the brain
  (chunked, embedded, searchable immediately — no separate ingest/sync step, no
  local-file dependency). KB wikis are read-only reference.
- **Security: standing grant bound to the writer (#40).** A `/security-review`
  found the initial standing grant was keyed by thread only — in a trusted
  multi-user channel, one user's approval auto-passed *every other user's*
  approval-tier writes in that thread for 8h. The grant is now keyed on
  `(thread, writer)`: a different user in the same thread still cards; the same
  writer stays covered.

### Internal

- **Removed `open_kb`** — the last KB tool that read local markdown at runtime
  (it `readFileSync`'d a KB's repo index and steered the agent to
  `Read`/`Grep`/`Glob` the wiki on disk). Its capability is fully covered by the
  DB-backed `kb_list_pages` / `kb_get_page` / `kb_search`. Every KB tool now
  sources purely from the gbrain DB. (`src/knowledge/mcp-tools.ts`,
  `src/soul/loader.ts`.)
- Tests across `brain-gate`, `brain-mcp-tools`, `soul`, `kb-mcp-tools`; finding
  + CLAUDE.md index marked RESOLVED. Decisions captured: `slaude_knowledge` /
  `/ingest` kept dormant (operator configures KBs read-only); `/1on1` memoize is
  personal by design (Mode C); durability (DB backup) and per-source ownership
  RBAC recorded as future direction.
