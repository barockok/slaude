## v0.16.1 — cron `target` + per-job passive mode

A small follow-up to v0.16.0's cron changes. v0.16.0 made cron fire unconditionally even when a human is active, and v0.15.0 added channel-vs-thread posting — but the `add_cron_job` MCP tool exposed neither: it always created thread jobs and there was no way to opt back into "wait for the human." This release closes both gaps and adds a per-job passive switch.

### Features

- **Cron: `target` + `when_active` on `add_cron_job`.** The MCP tool gained two optional params: `target` (`thread` default | `channel`) — previously channel posting existed only on the slash path — and `when_active` (`fire` default | `skip`). `skip` re-introduces "defer this run while a human is active in the target," now selectable per job instead of being a global behavior. Both list renderers and create confirmations surface `[target, passive]`.
- **Slash `/cron-add` passive flag.** Grammar is now `/cron-add "<expr>" "<prompt>" [channel|thread] [passive]` — flags optional, any order. `passive` maps to `when_active=skip`.

### Internal

- New `cron_jobs.when_active` column (`ALTER TABLE` migration, default `'fire'`) + `CronJob.whenActive`. Existing jobs are unaffected (default fire).
- The scheduler re-checks `isLive` only for `when_active='skip'` jobs; `fire` jobs always run. `#running` re-entry guard unchanged.
- Tests: DB defaults/persistence for `whenActive`, scheduler passive-skip-when-live, slash `passive` + combined `channel passive` parsing.
- Security review: no findings — both attributes are enum-constrained on an already-authorized, parameterized path; manager/approver gates intact; `target` only picks thread-vs-channel-root within the already-authorized channel (no arbitrary redirect).
- 746 tests pass, tsc clean.
