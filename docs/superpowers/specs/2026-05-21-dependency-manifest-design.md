# slaude Dependency Manifest — Design Document

**Status**: Draft v0.1
**Author**: Zidni Mubarok
**Date**: 2026-05-21
**Project**: [slaude](https://github.com/barockok/slaude)

---

## 1. Goal

Give slaude a semantic, declarative way to pin and install the agent's third-party dependencies as part of the image build:

1. Single manifest `slaude.json` declares the agent's external dependencies across three surfaces: **Claude Code plugins**, **skills**, and **knowledge bases**.
2. Lockfile `slaude.lock` pins every dependency to a git sha for reproducibility.
3. Build-time installer (`slaude install`) clones each dependency into the path the runtime expects. The image ships self-contained; runtime does no network resolution.
4. Claude Code's native loader picks up plugins / skills from `$CLAUDE_CONFIG_DIR`. Slaude itself doesn't reimplement plugin loading.
5. Knowledge bases follow Karpathy's LLM-wiki framework — plain markdown wikis the agent navigates with native `Read` / `Grep` / `Glob`. Slaude ships a thin `slaude_kb` MCP server that aggregates installed KBs by label.

### Non-goals

- **No MCP entries in the manifest.** `~/.slaude/mcp.json` continues to be the source of truth for MCP servers. The two systems compose at runtime but are declared separately. (MCP servers bundled inside CC plugins via `plugin.json` keep working through Claude Code's native plugin loader.)
- **No runtime resolver.** Dependencies are resolved at image build, never at agent boot. The container does not need network access for deps.
- **No registry / no central index.** v1 ships with git URLs only. A registry can be layered later by mapping short names to git URLs without changing the resolver shape.
- **No SOUL.md or memory in the manifest.** Identity and memory remain per-deploy, not packaged.
- **No KB embedding stack inside slaude.** Each KB is a markdown wiki; navigation is the LLM's job.

## 2. Manifest Shape

### 2.1 `slaude.json`

Operator-authored. Lives at `$SLAUDE_HOME/slaude.json`. Three top-level array sections.

```json
{
  "plugins": [
    {
      "marketplace": "github:anthropics/claude-plugins-official",
      "plugin": "superpowers",
      "ref": "5.1.0"
    },
    {
      "marketplace": "github:anthropics/claude-plugins-official",
      "plugin": "skill-creator",
      "ref": "5.1.0"
    },
    {
      "marketplace": "github:wshobson/claude-mem",
      "plugin": "claude-mem",
      "ref": "v1.0.0"
    }
  ],
  "skills": [
    {
      "git": "github:barockok/skill-release-notes",
      "ref": "v1.2.0"
    },
    {
      "git": "github:barockok/skill-runbook-search",
      "ref": "main",
      "slug": "runbook"
    }
  ],
  "knowledge": [
    {
      "label": "amartha-runbooks",
      "git": "github:amartha/runbooks-wiki",
      "ref": "v3.0.0"
    },
    {
      "label": "tools-cookbook",
      "git": "github:barockok/cookbook",
      "ref": "main"
    }
  ]
}
```

**Field rules:**

| Surface | Required | Optional |
|---|---|---|
| `plugins[]` | `marketplace`, `plugin`, `ref` | — |
| `skills[]` | `git`, `ref` | `slug` (defaults to last path segment of git URL, lowercased) |
| `knowledge[]` | `label`, `git`, `ref` | — |

**Git URL forms accepted** (all three sections):
- `github:owner/repo` — sugar for `https://github.com/owner/repo.git`
- `https://...` / `git@...:.../...git` — passed through verbatim
- No support for local paths in v1 (operators wanting a local skill drop it into `~/.slaude/skills/<slug>/` directly, bypassing the manifest)

**`ref` rules:**
- Tag (`v1.2.0`), branch (`main`), or sha (`abc1234…`). All accepted at declaration.
- Installer always **resolves to a sha** before writing the lockfile. Branch refs are intentionally allowed (for "always latest main") but the lockfile records the sha you got at install time.

### 2.2 `slaude.lock`

Auto-generated by `slaude install`. Operator does not edit by hand. Should be checked into the repo so image builds are reproducible.

```json
{
  "version": 1,
  "generated_at": "2026-05-21T11:00:00Z",
  "marketplaces": {
    "github:anthropics/claude-plugins-official@5.1.0": {
      "sha": "a1b2c3d4...",
      "plugins": {
        "superpowers":   { "version": "5.1.0", "subdir": "plugins/superpowers" },
        "skill-creator": { "version": "5.1.0", "subdir": "plugins/skill-creator" }
      }
    },
    "github:wshobson/claude-mem@v1.0.0": {
      "sha": "f9e8d7c6...",
      "plugins": {
        "claude-mem": { "version": "1.0.0", "subdir": "plugins/claude-mem" }
      }
    }
  },
  "skills": {
    "release-notes": { "git": "github:barockok/skill-release-notes", "ref": "v1.2.0", "sha": "..." },
    "runbook":       { "git": "github:barockok/skill-runbook-search", "ref": "main",   "sha": "..." }
  },
  "knowledge": {
    "amartha-runbooks": { "git": "github:amartha/runbooks-wiki", "ref": "v3.0.0", "sha": "..." },
    "tools-cookbook":   { "git": "github:barockok/cookbook",     "ref": "main",   "sha": "..." }
  }
}
```

Marketplaces are deduped by `(marketplace, ref)` so one git clone serves many plugin entries from the same marketplace at the same ref. Plugin `version` comes from the marketplace's `marketplace.json` entry, not from the git ref.

For **index-only marketplaces** (whose plugins live in external repos — see §4.4), each plugin entry under `plugins` additionally carries `source_repo` and `source_sha` so the lockfile pins both the marketplace and the per-plugin repo:

```json
"plugins": {
  "foo": { "version": "1.0.0", "subdir": "plugins/foo",
           "source_repo": "github:owner/foo", "source_sha": "abc1234..." }
}
```

## 3. Install Layout

`slaude install` writes into the following paths. All paths are inside `$SLAUDE_HOME` so they ride on the same persistent volume as the rest of slaude's state.

```
$SLAUDE_HOME/
├─ slaude.json              ← operator-authored
├─ slaude.lock              ← generated, committed
├─ .claude/                 ← CLAUDE_CONFIG_DIR (per existing memory entry)
│   └─ plugins/
│       └─ cache/
│           └─ <marketplace>/
│               └─ <plugin>/
│                   └─ <version>/
│                       └─ <plugin files…>          ← CC native loader picks up
├─ skills/
│   ├─ release-notes/
│   │   └─ SKILL.md         ← cloned-in skill
│   └─ runbook/
│       └─ SKILL.md
├─ knowledge/
│   ├─ amartha-runbooks/
│   │   ├─ README.md        ← entry point
│   │   ├─ topics/…
│   │   └─ …                ← KB author owns layout
│   └─ tools-cookbook/
│       └─ …
└─ mcp.json                 ← unchanged; outside the manifest
```

**Notes:**

- Plugins land in CC's native cache layout (`<config>/plugins/cache/<marketplace>/<plugin>/<version>/`) so Claude Code's existing plugin loader sees them with zero slaude-side runtime code.
- Skills are cloned one repo per skill. A repo wanting to ship many skills should publish itself as a CC plugin (which natively contains a `skills/` dir).
- KB clones are bare-cloned recursively; nothing is processed, indexed, or transformed. The wiki ships exactly as the author wrote it.

## 4. Installer (`slaude install` CLI)

New file: `src/cli/install.ts`. New `bun run install` script. Designed to run inside `Dockerfile` as a build step.

### 4.1 Algorithm

```
1. Read $SLAUDE_HOME/slaude.json. If missing, exit 0 (no deps declared).
2. Validate against zod schema. Fail fast on schema errors.
3. Load $SLAUDE_HOME/slaude.lock if present.
4. For each (marketplace, ref) pair in plugins[] (deduped):
     - If lock entry exists AND clone exists at expected path AND sha matches → skip.
     - Else: clone the marketplace. Clone strategy depends on ref shape:
       - Tag or branch → `git clone --depth 1 --branch <ref>`.
       - 40-char sha → `git clone --filter=tree:0 --no-checkout` + `git checkout <sha>`.
     - Read marketplace.json. For each requested plugin:
       - Self-contained subdir → copy/symlink the subdir into CC cache layout.
       - External source (`source.repo`, `source.ref`) → second clone (same strategy)
         into CC cache layout, capture the plugin's own sha for the lockfile.
5. For each entry in skills[]:
     - Compute slug (entry.slug || derive from git URL).
     - If lock entry exists AND clone exists AND sha matches → skip.
     - Else: clone into $SLAUDE_HOME/skills/<slug>/, capture sha.
     - Validate that cloned dir contains SKILL.md at root; warn otherwise.
6. For each entry in knowledge[]:
     - If lock entry exists AND clone exists AND sha matches → skip.
     - Else: clone into $SLAUDE_HOME/knowledge/<label>/, capture sha.
     - Validate that cloned dir contains README.md or index.md; warn otherwise.
7. Write updated $SLAUDE_HOME/slaude.lock atomically (write to .tmp, rename).
8. Print summary: N plugins / M skills / K KBs installed/updated/skipped.
```

### 4.2 Flags

- `slaude install` — default. Honour the lockfile when present; only resolve fresh for entries the lock doesn't cover.
- `slaude install --update` — ignore lockfile; resolve every entry fresh from its declared ref; rewrite the lockfile. Use this to pick up new commits on branch refs.
- `slaude install --frozen` — fail if any declared entry is not satisfied by the lockfile. Use in CI / image build to guarantee no network resolution.
- `slaude install --check` — exit 0 if lock satisfies manifest, exit 1 otherwise. No writes. For CI smoke tests.

### 4.3 Error model

- Schema invalid in `slaude.json` → exit 2, print zod error path.
- Git clone fails (network, auth, ref not found) → exit 3, print which entry failed. No partial lockfile write.
- `marketplace.json` missing from a cloned marketplace, or requested plugin name absent → exit 4, print marketplace path + available plugin names.
- Clone succeeds but SKILL.md / README.md missing → warn (stderr), do not fail. Operator may have a valid reason; the runtime will skip the empty surface.

### 4.4 Marketplace resolution

A marketplace repo's `marketplace.json` is the source of truth for what plugins it contains. Two shapes accepted:

- **Self-contained marketplace** — `marketplace.json` declares plugins as subdirs of the same repo:
  ```json
  { "plugins": [{"name": "superpowers", "version": "5.1.0", "path": "plugins/superpowers"}] }
  ```
- **Index-only marketplace** — `marketplace.json` references external repos per plugin:
  ```json
  { "plugins": [{"name": "foo", "source": {"type": "github", "repo": "owner/foo", "ref": "v1.0.0"}}] }
  ```
  Installer performs a second clone for each external-source plugin.

If a marketplace declares the same plugin via both shapes, the self-contained path wins.

## 5. Runtime — Plugins & Skills

**Zero new runtime code for plugins and skills.** Both are picked up by mechanisms that already exist:

- Plugins: Claude Code's native loader reads `$CLAUDE_CONFIG_DIR/plugins/cache/…`. The SDK child process inherits `CLAUDE_CONFIG_DIR` per the existing memory entry, so plugins installed by `slaude install` are visible without slaude-side changes.
- Skills: `src/skills/loader.ts:discoverSkills()` already scans `$SLAUDE_HOME/skills/` per inbound Slack message. Installed skills appear automatically.

Hot-reload semantics are unchanged. If an operator runs `slaude install --update` against a live container, the next inbound Slack turn picks up new skills via `discoverSkills()`. Plugins require an agent restart because Claude Code's plugin loader runs at SDK process start.

## 6. Runtime — Knowledge Bases

### 6.1 `slaude_kb` in-process MCP server

New file: `src/knowledge/mcp-tools.ts`. New file: `src/knowledge/loader.ts`. Mounted via the existing `McpResolver` alongside `slaude_slack` / `slaude_skills` / `slaude_session`.

**Two tools, both auto-allowed by the permission gate (read-only):**

| Tool | Args | Returns |
|---|---|---|
| `list_kbs` | `{}` | JSON array `[{label, description, path, index_file}]`. `description` = first non-empty line of `README.md` or the `description:` frontmatter key; truncated to 200 chars. `index_file` = `"README.md"` or `"index.md"`, whichever exists (README preferred). |
| `open_kb` | `{label: string}` | Full contents of the KB's index file as text. Error if label unknown. |

### 6.2 Discovery

`loadKbs()` scans `$SLAUDE_HOME/knowledge/`. For each subdir:
- Label = directory name (this is what the operator declared in `slaude.json`).
- Index file resolution: `README.md` → `index.md` → first `.md` alphabetically.
- Description: parse YAML frontmatter from index; if `description:` key present, use it; else use first non-blank, non-heading prose line.

Discovery runs lazily on first `list_kbs` / `open_kb` call per agent process; results cached for the process lifetime. (KB contents change at install time, not runtime, so no hot-reload story is needed.)

### 6.3 Why no built-in retrieval

KBs follow Karpathy's LLM-wiki framework: a structured markdown wiki where pages link to each other and the LLM navigates by reading. The agent uses Claude Code's native `Read` / `Grep` / `Glob` to follow links, scan headings, and pull relevant pages. Slaude does not pick an embedding model, does not maintain a vector store, does not chunk text. The wiki author owns the structure; the LLM is the search engine.

This also keeps the embedding-stack story singular: the guardrails design owns embedding decisions (EmbeddingGemma 300M + sqlite-vec). KBs deliberately stay out.

### 6.4 Soul-baseline integration

Add a short `## Knowledge bases` section to `RUNTIME_BASELINE` in `src/soul/baseline.ts`:

> You have access to one or more knowledge bases — markdown wikis you can read.
> Use `mcp__slaude_kb__list_kbs` to discover what's available, then
> `mcp__slaude_kb__open_kb` to load the entry page of a relevant KB. From there,
> navigate with `Read` / `Grep` / `Glob` on files under the returned `path`.
> Use this when the answer plausibly lives in operator-curated reference material
> rather than your own training.

## 7. Docker / Build Integration

`slaude install` is designed to run during image build, after dependencies are installed and before the runtime stage starts:

```dockerfile
# Build stage
FROM oven/bun:1 AS builder
WORKDIR /opt/slaude
COPY package.json bun.lock ./
RUN bun install --frozen-lockfile
COPY . .

# Install agent dependencies declared in slaude.json
ENV SLAUDE_HOME=/opt/slaude/.slaude
COPY .slaude/slaude.json .slaude/slaude.lock $SLAUDE_HOME/
RUN bun run cli install --frozen

# Runtime stage
FROM oven/bun:1-slim
ENV SLAUDE_HOME=/data/.slaude \
    CLAUDE_CONFIG_DIR=/data/.slaude/.claude
COPY --from=builder /opt/slaude /opt/slaude
COPY --from=builder /opt/slaude/.slaude/skills    /data/.slaude/skills
COPY --from=builder /opt/slaude/.slaude/knowledge /data/.slaude/knowledge
COPY --from=builder /opt/slaude/.slaude/.claude   /data/.slaude/.claude
# slaude.json + slaude.lock + mcp.json + SOUL.md mounted from PVC at runtime
CMD ["bun", "src/server.ts"]
```

Two layout choices the spec deliberately makes:

- **`slaude install --frozen` inside the image build** — guarantees the image is reproducible and never reaches out to git at runtime.
- **Install output (`.claude/plugins/`, `skills/`, `knowledge/`) baked into the image, NOT mounted from the PVC.** Otherwise every deploy that bumps a plugin version needs a PVC dance. The PVC keeps operator-authored files (`SOUL.md`, `mcp.json`, `slaude.json`, `slaude.lock`) and runtime state (`db.sqlite`, `cache/`, `workspaces/`). Plugins / skills / KBs are image artifacts.

(`slaude.json` and `slaude.lock` live on the PVC so an operator can edit + redeploy. The next image build picks up the new lockfile and bakes in updated artifacts.)

## 8. Testing

Following the project's existing patterns (`tests/*.test.ts`, `bun test --coverage` ≥ 97% line, ≥ 80% func):

| Test file | Covers |
|---|---|
| `tests/manifest-schema.test.ts` | zod schema accepts all valid shapes (3 sections × form variants), rejects malformed entries (missing required fields, bad git URLs, duplicate labels). |
| `tests/install-resolver.test.ts` | Pure resolver logic: marketplace dedup, slug derivation, lockfile-hit detection. No git calls — git operations stubbed at a single `gitOps` interface. |
| `tests/install-lockfile.test.ts` | Lockfile read / write / atomic-replace; `--frozen` rejection when manifest entries unsatisfied; `--check` exit codes. |
| `tests/kb-loader.test.ts` | `loadKbs()` discovery: README precedence, frontmatter parsing, description-truncation. Tmp dirs with fixture wikis. |
| `tests/kb-mcp-tools.test.ts` | `list_kbs` / `open_kb` handler shape, unknown-label error, JSON-stringify cleanliness. |

Integration smoke (not coverage-gated, run manually):
- `bun run cli install --update` against a tiny fixture marketplace + fixture skill + fixture KB. Asserts files land in the right paths and lockfile shape round-trips.

## 9. Migration

For existing slaude deploys with no `slaude.json`:

- `slaude install` with no manifest exits 0. Nothing breaks.
- `mcp.json` is untouched by anything in this design.
- Operators who already maintain custom skills under `~/.slaude/skills/` keep them working — those are local skills, not declared in the manifest, and `discoverSkills()` doesn't care where they came from.
- Migration path for an existing custom skill into the manifest: push it to its own git repo, add an entry to `slaude.json`, run `slaude install --update`, delete the local copy.

No breaking changes to any existing config file or runtime behaviour.

## 10. Open / Deferred

Tracked for future iterations, intentionally out of scope here:

- **Registry / short names.** v1 = git URLs only. A `slaude.dev/registry.json` mapping `superpowers` → `github:anthropics/claude-plugins-official#superpowers` is mechanical to add later.
- **Local-path entries.** Out of v1; operators with local skills drop into `~/.slaude/skills/` directly. Add `{path: "..."}` later if demand shows up.
- **`slaude search` / `slaude list`.** CLI introspection (what's installed, what's outdated). Useful once the install command lands.
- **HTTP / SSE knowledge sources.** v1 = git wikis only. A `kind: "http"` entry could fetch a remote wiki at install time, but no concrete use case yet.
- **Multi-marketplace plugin name collisions.** If two marketplaces both ship a plugin called `superpowers`, the second one clobbers the first at the cache path. Defer until it actually happens; v1 strategy is "last-write wins, log warning."
- **Auto-update / cron.** Operators run `slaude install --update` manually for now. A scheduled drift detector is plausible later.
